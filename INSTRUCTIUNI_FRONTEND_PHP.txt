SMARTKINETO - varianta cu frontend legat la PHP + MySQL
========================================================

CE S-A SCHIMBAT
---------------
1. Frontendul din folderul frontend/ nu mai folosește datele demo ca sursă principală.
2. Am adăugat folderul api/, care conține fișiere PHP ce returnează JSON.
3. frontend/script.js comunică acum cu PHP prin fetch(), de exemplu:
   - api/login.php
   - api/me.php
   - api/data.php
   - api/book_session.php
   - api/cancel_booking.php
   - api/update_profile.php
4. Baza de date rămâne MySQL/MariaDB, prin Database.php și init.php.

ARHITECTURA PE CARE O PREZINȚI PROFULUI
---------------------------------------
Frontend JS <-> PHP API <-> MySQL

Frontendul face doar afișarea și interacțiunea.
PHP-ul primește cereri, verifică sesiunea, lucrează cu baza de date și trimite JSON.
MySQL păstrează utilizatori, abonamente, programări, săli, istoricul activităților.

CE TREBUIE SĂ ARĂȚI PROFULUI
----------------------------
1. Deschizi frontend/index.html și arăți că interfața este partea vizuală.
2. Deschizi frontend/script.js și arăți că API_BASE este:
   const API_BASE = "../api/";

3. Arăți funcția Api.request(), care face fetch către PHP.
4. Arăți Auth.loginAsync(), care trimite email/parolă către api/login.php.
5. Arăți api/login.php, unde PHP caută userul în tabela users și verifică parola.
6. Arăți api/data.php, care ia din MySQL utilizatori, abonamente, săli, programări și le trimite către frontend.
7. Arăți api/book_session.php, care salvează o rezervare reală în tabela appointments.
8. Arăți api/cancel_booking.php, care marchează rezervarea ca anulată în baza de date.

CE SPUI DACĂ TE ÎNTREABĂ DE CE EXISTĂ ȘI dashboard.php
------------------------------------------------------
Spui așa:

În proiect a existat inițial o variantă clasică PHP, unde fiecare pagină era generată direct de PHP, precum login.php, dashboard.php și paginile din pages/.
Ulterior s-a realizat o interfață frontend mai modernă în HTML/CSS/JavaScript, în folderul frontend/.
Pentru integrare, am păstrat frontendul ca interfață principală și am transformat PHP-ul într-un backend/API care returnează JSON.
Astfel, nu mai sunt două aplicații separate, ci frontendul comunică direct cu PHP-ul prin fetch(), iar PHP-ul comunică mai departe cu MySQL.

dashboard.php poate rămâne ca variantă veche/clasică, dar pentru prezentare folosim frontend/index.html conectat la api/.

CE SPUI DESPRE localStorage
---------------------------
Spui așa:

În versiunea inițială, localStorage era folosit pentru date demo. În versiunea integrată, datele vin din baza de date prin api/data.php, iar Store-ul din JavaScript este folosit doar ca memorie temporară pentru afișare în interfață. Sursa reală a datelor este baza de date MySQL.

FLUX LOGIN
----------
1. Userul scrie email și parolă în frontend.
2. JavaScript trimite datele cu fetch() la api/login.php.
3. PHP caută utilizatorul în tabela users.
4. PHP verifică parola cu password_verify().
5. Dacă e corect, PHP salvează user_id, email, username și role în $_SESSION.
6. PHP returnează JSON cu datele utilizatorului.
7. Frontendul afișează meniul în funcție de rol.

FLUX ÎNCĂRCARE DATE
-------------------
1. După login, script.js apelează loadAppData().
2. loadAppData() cere api/data.php.
3. api/data.php citește datele din MySQL.
4. Frontendul salvează temporar datele în Store.
5. Funcțiile renderAdminDashboard(), renderMemberDashboard(), renderSchedule() etc. afișează datele.

FLUX REZERVARE
--------------
1. Membrul apasă „Rezervă loc”.
2. bookSession() din script.js trimite session_id către api/book_session.php.
3. PHP verifică dacă userul este logat.
4. PHP verifică dacă are abonament activ.
5. PHP inserează rezervarea în tabela appointments.
6. PHP adaugă și o intrare în activities_history.
7. Frontendul reîncarcă datele din baza de date și actualizează ecranul.

FIȘIERE NOI IMPORTANTE
----------------------

api/cancel_booking.php  - anulează rezervare.
api/update_profile.php  - salvează modificări profil.
set_demo_passwords.sql  - setează parole cunoscute pentru conturile demo.
