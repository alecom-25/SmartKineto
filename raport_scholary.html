1. Introducere

1.1 Scop

Scopul acestui document este de a detalia cerințele software, arhitectura și logica de business pentru aplicația web SmartKineto.

1.2 Convenții ale Documentului

  • UI/UX: User Interface / User Experience.
  • State-based UI: Interfață care își schimbă afișajul strict pe baza stării datelor persistente din server.
  • ACID: Atomicitate, Consistență, Izolare, Durabilitate (în contextul bazelor de date).

1.3 Scopul Produsului

SmartKineto este o platformă digitală concepută pentru a optimiza fluxurile de lucru dintr-o clinică modernă. Aceasta elimină suprapunerile de programări, automatizează comunicarea prin email și oferă o gestiune transparentă a abonamentelor și utilizatorilor pe baza unor roluri predefinite.

2. Descriere Generală

2.1 Perspectiva Produsului și Arhitectura

Sistemul urmează o arhitectură Client-Server, iar backend-ul este construit în PHP și interacționează cu o bază de date relațională MySQL. Comunicarea asincronă cu interfața se realizează prin API-uri interne (AJAX) pentru a oferi un răspuns imediat fără reîncărcarea paginii.

2.2 Clase de Utilizatori

Sistemul gestionează trei tipuri de actori, fiecare cu un nivel de acces strict definit:

  • Administrator: Acces total asupra sistemului, export de date (CSV/XML), gestiunea staff-ului, a membrilor și a sălilor disponibile.
  • Staff (Antrenor / Kinetoterapeut): Gestiunea propriului orar, capabilitatea de a aproba, reprograma sau anula ședințe și de a trimite mesaje personalizate pacienților, precum și monitorizarea și eventual aprobarea sau respingerea unei schimbări de abonament.
  • Membru: Vizualizarea istoricului plăților, a stării abonamentului și inițierea cererilor de programare pe baza sloturilor disponibile în timp real.

2.3 Proiectarea Interfeței Web (UI/UX)

Interfața web a fost proiectată conform principiilor Responsive Web Design, utilizând HTML și CSS (fără a depinde excesiv de framework-uri externe masive).

Motivarea alegerilor de design:

  • S-a optat pentru un layout minimalist și aerisit, cu o paletă de culori axată pe albastru (profesionalism/medical) și culori semantice (verde pentru succes, roșu pentru erori).
  • Formularele și tabelele utilizează efecte de umbrire (box-shadow) și carduri pentru a focaliza atenția utilizatorului pe fluxul de acțiune principal (ex: efectuarea unei rezervări).

3. Caracteristicile Sistemului (Funcționalități Esențiale)

3.1 Motorul Dinamic de Programări

Sistemul gestionează rezervările într-un mod inteligent. Nu doar interoghează baza de date pentru a exclude intervalele ocupate, ci implementează o validare contextuală: dacă un pacient solicită o programare pentru ziua curentă, algoritmul compară ora serverului cu sloturile disponibile și ascunde dinamic orele deja expirate, prevenind posibilitatea unor rezervări în trecut.

3.2 Notificări Persistente

Pentru alertarea pacienților cu privire la stadiul programării, a fost implementat un sistem robust de notificări. În loc de stocarea alertelor în memoria volatilă a browserului (sesiuni), s-a introdus flag-ul is_read la nivel de rând în tabelul appointments. Astfel, "închiderea" unei notificări reprezintă un UPDATE direct în baza de date, asigurând persistența acțiunii pe orice dispozitiv.

3.3 Comunicare Automatizată prin Email cu Mesaje Personalizate

Sistemul a fost integrat cu protocolul SMTP utilizând librăria PHPMailer. La fiecare acțiune critică (Confirmare, Reprogramare, Anulare), pacienții primesc șabloane HTML responsive. În plus, arhitectura permite terapeuților să injecteze mesaje personalizate (ex: indicații medicale) care sunt concatenate automat în structura email-ului.

3.4 Gestiunea Tranzacțiilor SQL

Operațiunile care afectează simultan mai multe tabele (ex: adăugarea unui nou utilizator afectează users, user_details și subscriptions) sunt împachetate în Tranzacții SQL (utilizând $db->beginTransaction() și commit()). Aceasta garantează atomicitatea procesului, prevenind coruperea datelor în caz de eroare fatală la execuție.

3.5 Gestiunea Avansată a Utilizatorilor și a Abonamentelor

Sistemul integrează un modul complex de autentificare și autorizare. Administratorul poate asigna niveluri de acces (roles) și fiecare membru își poate alege un abonanemt (Fitness, Forță, Kineto, mixte și VIP), schimbare ce trebuie confirmată de antrenor. Informațiile sunt filtrate dinamic pe baza acestor roluri (ex: un pacient fără abonament fitness nu va putea selecta anumite tipuri de servicii sau antrenori).

3.6 Afișarea Dinamică a Orarului (AJAX)

Pentru o experiență de utilizare fluidă, selectarea unei date în formularul de rezervare declanșează o interogare asincronă (AJAX) către server. Aceasta returnează în format JSON doar antrenorii și sălile disponibile exact în acel moment, fără a necesita reîncărcarea întregii pagini.

4. Structura Datelor (System Design)

Proveniența și modelarea datelor respectă principiile formelor normale din bazele de date relaționale. Integritatea referențială este asigurată prin constrângeri FOREIGN KEY (cu reguli ON DELETE CASCADE / SET NULL).

Structura de bază implică separarea conceptuală:

  • Date de Autentificare: Tabelul users (stochează doar email, parolă hash-uită și rol).
  • Date Personale: Tabelul user_details (nume, telefon, adresă), legat 1:1 de `users`.
  • Gestiunea Operațională: Tabelele appointments, rooms, session_types, interconectate pentru a genera disponibilitatea reală.

Aplicația expune și consumă date prin intermediul unor micro-scripturi interne (ex: get_slots.php, get_rooms_availability.php) care funcționează ca un API intern, returnând date exclusiv în format JSON către interfața clientului (via fetch() în JavaScript).

5. Cerințe Non-funcționale

5.1 Securitate

  • Toate parolele sunt securizate unidirecțional folosind funcția password_hash() (algoritm BCRYPT).
  • Protecția împotriva injecțiilor SQL este garantată prin folosirea exclusivă a obiectelor PDO cu declarații preparate (Prepared Statements).
  • Validarea se face pe două straturi: în frontend (atribute HTML) și în backend (verificări PHP înainte de interacțiunea cu baza de date).

5.2 Interoperabilitate

Sistemul include module de import și export al datelor (Staff și Rapoarte) în formate standardizate (CSV și XML), permițând importul facil în alte sisteme ERP sau programe de calcul tabelar.
